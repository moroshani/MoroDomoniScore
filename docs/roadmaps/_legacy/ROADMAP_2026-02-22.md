## Strategic Plan A: Full JS Stack Migration (`Next.js` + Node.js)
Goal: migrate from `Vite + React + Express API` to a unified modern JS stack while protecting current production behavior.

### A1) Decision and Architecture Targets
- Frontend target: `Next.js 16` (App Router, React 19), Persian RTL-first.
- Runtime target: Node.js LTS line on VPS (single JS runtime strategy).
- Data layer: keep Prisma + PostgreSQL.
- Realtime: keep current sync server initially; evaluate migration later.
- Deployment model: phased parallel run, not big-bang rewrite.

### A2) Migration Scope Rules
- No regression on password auth flows.
- No regression on gameplay scoring logic or Quick Night behavior.
- PWA behavior must remain aligned between installed and browser modes.
- Existing API contract stays source-of-truth until final cutover.

### A3) Phased Execution Plan
1. Phase 0: Contract Freeze and Baseline (3-5 days)
- Freeze payload shapes for `/api/auth`, `/api/users`, `/api/players`, `/api/history`.
- Add contract smoke tests for all critical endpoints.
- Capture route parity checklist for all user-facing pages.

2. Phase 1: Next App Bootstrap (3-4 days)
- Create Next app in parallel (`apps/morodomino-web` style structure).
- Configure RTL/Persian typography, routing shells, error boundaries.
- Keep `/api/*` served by current Express during this phase.

3. Phase 2: Auth/Session Port (4-7 days)
- Port login/register/me UI with current backend.
- Keep password auth/session UX as source-of-truth.

4. Phase 3: Non-Gameplay Routes (5-8 days)
- Port Guide, History, Stats, Profile, Settings.
- Keep existing APIs and DB as-is.
- Validate responsive behavior and Persian copy parity.

5. Phase 4: Gameplay Core Port (7-12 days)
- Port Play flow, mode selection, teams, rounds, recap, audit log.
- Preserve quick-night non-persistence behavior exactly.
- Validate with fixture-based scoring comparisons.

6. Phase 5: PWA Re-Hardening (3-5 days)
- Re-implement SW/manifest/offline behavior on Next build pipeline.
- Verify install/update/offline parity on Android/iOS/desktop.

7. Phase 6: Cutover (2-4 days)
- Nginx switch to Next web upstream.
- Keep old UI rollback route for one release window.
- Post-cutover smoke run (auth/passkey/game/history/stats/spectator).

8. Phase 7: Optional API Consolidation (2-6 weeks, optional)
- Migrate selected Express handlers into Next route handlers gradually.
- Keep WS sync separate until proven replacement is ready.

### A4) Quality Gates (Must Pass Before Final Cutover)
- Auth/password/passkey flows all pass in real device tests.
- PWA checks pass: manifest, SW, offline fallback, update flow.
- Scoring regression pack: no behavior drift in round/set/night logic.
- Live spectator sync remains stable under concurrent viewers.
- Docs/runbook/ops commands updated and verified on VPS.

### A5) Risk Register and Mitigation
- Risk: passkey regressions during framework move.
Mitigation: keep backend auth unchanged through Phases 1-4; migrate UI first.
- Risk: PWA cache/update mismatch.
Mitigation: explicit SW versioning and staged rollout checks.
- Risk: gameplay logic drift.
Mitigation: fixture comparisons + API/event smoke scripts before cutover.
- Risk: ops complexity with dual stack.
Mitigation: strict phase ownership and reversible Nginx switching.

### A6) Non-Functional Targets (Required for Cutover)
- Availability target: 99.9% monthly for public app.
- Error budget policy: block feature cutovers when budget is exhausted.
- API p95 latency target: `<= 300ms` for core read/write endpoints under normal load.
- Web vitals targets:
  - mobile LCP `<= 2.5s`
  - mobile CLS `< 0.1`
  - INP `<= 200ms`
- Accessibility targets:
  - WCAG 2.2 AA baseline for critical flows
  - keyboard and screen-reader coverage for auth/game/profile/settings
- Security targets:
  - no critical/high unresolved vulnerabilities in production deploy window
  - passkey + password fallback verified on at least 2 Android devices, 2 iOS devices, 2 desktop browsers.

### A7) Data and Domain Evolution Strategy
- Keep existing Prisma schema as canonical source during migration.
- Introduce domain service boundaries without changing table contracts first.
- Add migration-safe data adapters:
  - `NightRecordAdapter`
  - `PasskeyAdapter`
  - `ProfileSecurityAdapter`
- Add versioned payload contracts for app-to-api communication to support rollback compatibility.
- Define schema evolution workflow:
  - additive changes first
  - dual-read/dual-write where necessary
  - destructive changes only after one stable release cycle.

### A8) Platform and DevEx Evolution
- Monorepo governance:
  - app packages (`web`, `api`, `sync`)
  - shared contracts and auth utilities
  - shared lint/tsconfig/build profiles.
- CI/CD evolution:
  - pipeline stages: lint -> typecheck -> unit -> integration -> build -> deploy -> smoke.
  - release channels: `staging`, `canary`, `stable`.
- Preview environments for major route migrations.
- Release train cadence:
  - weekly stable release window,
  - emergency patch workflow within 60 minutes.

### A9) QA, Test, and Release Governance
- Test pyramid:
  - unit tests for domain logic
  - contract/integration tests for API surfaces
  - E2E tests for auth/passkey/gameplay/PWA.
- Mandatory release checklist:
  - auth/password/passkey smoke
  - profile/settings smoke
  - history/stats/quick-night smoke
  - spectator sync smoke
  - manifest/sw/offline smoke.
- Regression suites:
  - deterministic gameplay fixture pack
  - cross-browser auth/passkey pack
  - installed-vs-browser parity pack.

### A10) Cost and Operations Planning
- Capacity planning:
  - baseline CPU/memory thresholds for API/sync/web services
  - alert thresholds and auto-restart policy validation.
- Observability:
  - centralized structured logs
  - request tracing for auth and gameplay paths
  - KPI dashboards (active users, error rate, passkey success rate).
- Backup/DR:
  - nightly backups with restore drills
  - target recovery objectives (RTO/RPO) documented and tested.

## Strategic Plan B: Admin App (`mdadmin.ariaprojectsdashboard.ir`)
Goal: create a separate operational/control application for support, security, data ops, and release management.

### B1) Product Boundary
- Separate app and subdomain: `mdadmin.ariaprojectsdashboard.ir`.
- Separate login surface and admin-only permissions.
- Shared database with strict RBAC and audit logs for every admin action.

### B2) Proposed Stack
- Next.js (same strategic stack as primary app migration target).
- Node.js runtime, Prisma, PostgreSQL.
- API integration pattern:
  - read/write via admin-scoped endpoints (`/api/admin/*`),
  - or proxy to core APIs with service credentials where appropriate.

### B3) Roles and Access Model
- `superadmin`: full control including destructive actions.
- `ops_admin`: infra/runtime and incident operations.
- `support_admin`: user support and account recovery tools.
- `readonly_auditor`: read-only compliance and reporting.

### B4) Admin Modules (What Will Be Included)
1. System Health and Runtime
- API/sync/nginx health, uptime, error rates, deployment version.

2. User and Security Operations
- user search (email/username/id),
- passkey/device list and revoke,
- password/login issue diagnostics,
- lockout and suspicious activity inspection.

3. Gameplay Data Operations
- history explorer, data consistency checks, controlled fixes.
- export tools for analytics/review.

4. Moderation and Abuse Controls
- high-failure login/passkey dashboards,
- IP/device anomaly indicators,
- protective throttling and temporary restrictions (policy-based).

5. Release and Configuration Center
- feature flag control,
- active version verification,
- SW rollout/cache status view,
- maintenance/incident messaging controls.

6. Audit and Compliance
- immutable admin action log (`who`, `what`, `when`, `why`),
- exportable audit reports.

### B5) Delivery Phases
1. Admin Phase A (2-3 weeks)
- auth/RBAC foundation, health dashboard, user lookup, audit log base.

2. Admin Phase B (3-5 weeks)
- passkey/user support toolkit, data ops utilities, moderation controls.

3. Admin Phase C (2-4 weeks)
- release center, flags, incident tooling, KPI dashboards.

### B6) Security and Operations Requirements
- Enforce MFA/passkey for admin accounts.
- CSRF/CORS locked to admin origin.
- Optional IP allowlist for privileged roles.
- Rate limits and lockout policies separate from public app.
- Full audit trail required for destructive actions.

### B7) Day-to-Day Usage Model
- Morning checks: service health, auth anomaly trends.
- Support workflow: find user, inspect session/passkeys, apply safe fixes.
- Incident workflow: identify failure domain, apply mitigations, publish status.
- Release workflow: verify deployed version and rollout state.

### B8) Admin Domain Model and Data Boundaries
- Keep admin writes explicit and auditable; no hidden side effects.
- Introduce admin-specific tables:
  - `AdminUser`
  - `AdminRoleBinding`
  - `AdminAuditLog`
  - `FeatureFlag`
  - `IncidentBanner`
  - `SupportCase` (optional).
- Enforce data minimization:
  - support role sees only required user attributes
  - destructive operations require stronger role and reason fields.

### B9) Security Architecture for Admin Surface
- Mandatory strong auth:
  - password + passkey (or equivalent MFA) for all admin roles.
- Session hardening:
  - short-lived tokens
  - idle timeout
  - step-up auth for destructive actions.
- Network hardening:
  - optional IP allowlist for privileged actions
  - strict CORS/CSRF policies
  - per-role rate limits.
- Secrets and key management:
  - rotate service credentials quarterly
  - prevent secret leakage in UI/logs/audit payloads.

### B10) Operational Playbooks in Admin App
- Incident center workflows:
  - detect -> triage -> mitigate -> communicate -> postmortem.
- Feature flag playbooks:
  - safe rollout
  - instant rollback
  - blast-radius metadata.
- Support playbooks:
  - passkey troubleshooting path
  - account recovery path
  - suspicious activity escalation.
- Data ops playbooks:
  - export requests
  - correction requests
  - retention and deletion workflows.

### B11) Governance, Compliance, and Audit
- Every privileged action must include:
  - actor,
  - timestamp,
  - target,
  - reason code,
  - before/after snapshot references.
- Add approval workflow for high-risk actions:
  - account hard-delete
  - bulk data modification
  - lockout policy override.
- Quarterly access review:
  - remove stale admins
  - verify least-privilege role assignments.

### B12) Adoption and Rollout Strategy
- Stage rollout:
  1. internal-only superadmin usage
  2. limited support team
  3. full ops + support + audit roles.
- Training requirements:
  - role-specific runbooks
  - incident drills
  - audit procedure drills.
- Success metrics:
  - MTTR reduction
  - support ticket resolution time
  - passkey issue diagnosis time
  - deployment confidence metrics.
